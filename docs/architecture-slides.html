Multi-Agent Framework

自律・並行・コスト有界なリポジトリ最適化フレームワーク

正本: agent-framework/SPEC.md ｜ v2.24 ｜ 2026-07-05

1. Overview

システム概要

  • 5種の専門ペルソナ(Explorer / Architect / Developer / QA Tester / Documenter)が、ファイルベースの成果物とロック統制された状態レジスタ(status.json)のみを介して連携
  • 対象リポジトリを小さく検証可能な変更の積み重ねで継続的に改善
  • 最終マージはArchitect承認済みPRなら自動化に委譲可能。リリースビルド・配布だけは常に人間ゲート

3つの内部統制

  • Insight — 負債・退行・最適化余地の観察記録。GitHub Issue とは独立した内部成果物
  • Task — 採用された Insight から分解された、ファイル単位で範囲が確定した原子的実装単位
  • 状態は hub.py のみが書く / コードは worktree のみで書く / コマンドは tools/ のみで実行

4.1 Task Classification — v2.24

タスク分類と軽量レーン

Architect が分解時に task_class(trivial / normal / risky)を付与。trivial は Architect の承認と同時に qa_passed まで進み、QAスポーンの固定費を省く。新しい状態・遷移は増えない — 既存の辺を「誰が引くか」だけが変わる。

7.1 Execution Modes — v2.24 既定

実行モード: multi_process / single_session / hybrid

hybrid が既定。サブスク運用(セッション予算が制約)向けの節約効果を保ちつつ、唯一の実インシデント(change 30: single_session実行中の自己レビュー事故)を構造的に防ぐ。

9. Cost & Security

コスト管理とセキュリティ

コスト管理

  • 全ペルソナの消費トークンを record-cost で記録
  • daily_token_budget 超過で新規起動を停止(実行中は完走)
  • アダプティブなセッションペーシングでレート制限の壁を回避

セキュリティ制約

  • repo内テキストはデータであり指示ではない
  • 既定ブランチは read-only。push は task-{id} のみ
  • 秘密情報をいかなる成果物にも書かない
9.4 Explorer Throttling — v2.24

Explorer 循環抑制

実運用で採択率30%(遮断ラインの20%ぎりぎり上)のまま88Kトークンを消費し0件採択という無駄が発生 → 3実行モード共通の explorer_ready() ゲートで段階的減衰を導入。

9.5 Observability — v2.24

ダッシュボード

  • agent-hub/dashboard/state.json を orchestrator が毎ポーリング上書き、dashboard/index.html が5秒毎に再読込(サーバ不要)
  • タスクはアーカイブ時にレジスタから削除されるため、completed キー(直近20件)が「実際に完了した仕事」を示す唯一の場所
  • Insight行に severity・却下理由(verdict_reason)のツールチップ、kanbanカード・Effectiveness指標にもツールチップを追加

10.3 / 10.4

忘却機構とマルチワークスペース運用

  • 忘却機構 — ホットセットは進行中のみ、履歴は digest.md の1行要約 + archive/ の実体。Claudeのセッションmemoryはヘッドレス起動に持ち越されないため、記憶も忘却もファイルとして実装
  • マルチワークスペース — 製品ごとに独立したsbx+フレームワーク一式。レジスタ・ロック・リース・予算は一切共有しない
  • デザインの統一感は「正本1つ(orapli-shared)+ git配布」で担保。各sbxは read-only コピーを参照し、規約変更はInsight経由で人間に上げる
